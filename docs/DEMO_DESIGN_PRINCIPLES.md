# Demo 设计原则：内化企业治理，不改写客户 PRD

更新日期：2026-07-20

## 1. 优先级

两份客户 PRD 是功能、业务对象、流程和验收的唯一需求来源。InsightFlow 与 Agent Matrix Infra 的材料只用于指导实现质量，不能据此新增客户没有提出的 Agent Runtime、Eval、知识平台、工具网关或其他产品模块。

发生冲突时按以下顺序处理：

1. 客户 PRD 中明确写出的业务目标与验收。
2. 用户在本轮明确提出的管理系统/大屏拆分和视觉要求。
3. 参考包的布局与交互方式。
4. 团队对企业治理和技术架构的内部理解。

## 2. 如何自然体现团队理解

团队理解不通过增加菜单或宣传文案体现，而通过现有功能的完整性体现：

- 资产目录：同一套资产数据在概览、列表、详情和大屏使用相同口径。
- 权限：角色决定能做什么，资产范围决定能对哪些数据做；所有页面复用同一判定模块。
- 责任：资产显示负责人和团队，审批显示提交人、处理角色、目标范围与等待时间。
- 生命周期：个人资产必须先进入团队，再进入通用范围，不能绕过中间责任节点。
- 留痕：创建、提交晋级、通过和驳回写入现有活动记录，用户能看到结果和时间。
- 运营：大屏同时呈现原 PRD 要求的规模、成功率、成本、在线、审批和风险信息，切换视角时统一过滤。
- 技术边界：Demo 已实现能力与生产级身份、服务端策略、事务和不可变审计的差距要如实说明。

## 3. 明确不做

- 不把客户的 AI 资产运营系统改造成 InsightFlow 或 Agent Matrix Infra 产品展示。
- 不为展示架构水平增加没有数据支撑的指标、对象或流程。
- 不把演示角色切换描述成生产鉴权。
- 不把前端多次写入描述成服务端事务。
- 不迁移到参考包的 Next.js 技术栈，也不复制参考包的无关业务页面。

## 4. 判断完成的方式

客户应先看懂原 PRD 的资产运营、权限和审批流程；继续操作时，能够感受到数据口径一致、权限边界明确、责任可追、异常可解释。即使不出现“Agent DevOps”或“控制面”等术语，也能证明团队理解企业系统如何长期管理。
