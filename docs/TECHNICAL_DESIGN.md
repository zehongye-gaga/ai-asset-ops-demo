# 技术方案：企业治理演示增强

版本：v1.1
更新日期：2026-07-19

## 1. 当前调用链

1. 浏览器加载 React 应用和 `/config.js`。
2. `src/lib/insforge.ts` 使用运行时地址和匿名 Key 创建 InsForge SDK 客户端。
3. `src/api.ts` 读取 `assets`、`approval_requests`、`activity_events`。
4. `src/App.tsx` 根据当前角色、范围和类型过滤数据，渲染概览、目录、审批和大屏。
5. 新建、提交晋级和审批写回 InsForge，刷新三张表后更新页面。

当前问题是权限判断散落在页面条件里，只有三个角色，且大屏没有独立布局和视角口径。继续堆条件会导致不同页面对同一角色给出不同结论。

用户提供的 InsightFlow 架构图只用于校准分层、职责、权限、审计和可观测性的表达方式，不改变本系统的业务对象和功能范围。

## 2. 目标结构

```text
src/
├── access-control.ts   # 角色、范围和动作的唯一判定入口
├── governance.ts       # 治理指标、风险和视角数据的纯函数
├── App.tsx             # 页面状态、交互编排和组件
├── api.ts              # InsForge 读写与活动留痕
├── types.ts            # 角色、资产、审批、事件类型
└── styles.css          # 白蓝企业视觉与全屏大屏
```

### 2.1 权限模块

`access-control.ts` 只接收可信的业务上下文：角色、资产范围、资产负责人和当前用户。它输出：

- 默认范围。
- 可创建的范围。
- 是否可管理资产。
- 是否可提交晋级。
- 是否可处理审批。
- 面向用户的权限说明。

所有页面必须调用这组函数，不再各自复制角色条件。这样角色矩阵只有一套权威逻辑，测试可以覆盖所有角色和范围组合。

### 2.2 治理指标模块

`governance.ts` 使用已经加载的三张表计算：

- 待审批与超时风险。
- 低成功率、离线和审核中资产数量。
- 资产类型、业务域、生命周期分布。
- 审计事件数量和治理覆盖说明。
- 全局/团队/个人视角的数据集合。

它不发请求、不修改状态，便于单元测试和大屏复用。

## 3. 角色与范围判定

### 3.1 角色

```text
system_admin  系统管理员
operator      运营人员
team_admin    团队管理员
personal      个人
```

### 3.2 操作判定

- 系统管理员可以管理所有资产，并可兜底处理全部审批。
- 运营人员管理通用资产和本人资产，处理团队→通用申请。
- 团队管理员管理当前团队“研发中心”的团队资产和本人资产，处理本团队个人→团队申请。
- 个人仅管理本人个人资产。
- 通用和团队资产对无管理权限的角色保持可见，但明确标记只读。

运营人员在通用范围仅可直接创建 Skill、知识库和 MCP；创建智能体或应用必须先落在个人/团队范围，再按晋级流程进入通用范围。该限制同样由 `access-control.ts` 判定。

当前 Demo 没有登录用户 ID，因此“本人”以现有演示用户“叶泽宏”为可信上下文。生产版必须从登录会话取得用户和团队，不允许由前端自由传入。

## 4. 数据与写入

### 4.1 保持现有表结构

本次继续使用：

- `assets`：资产主数据和运营指标。
- `approval_requests`：晋级申请与处理状态。
- `activity_events`：运营动态和关键操作留痕。

不新增 Migration，避免为了演示 UI 引入第二套治理数据源。

现有 `approval_requests.approver_role` 只允许 `team_admin` 和 `system_admin`。团队→通用申请继续使用 `system_admin` 作为“通用发布队列”的存储代码，前端允许运营人员处理该队列，系统管理员保留兜底权限。这样不改现有协议，也不会伪造数据库已经存在 `operator` 枚举。

### 4.2 活动留痕

以下动作必须写 `activity_events`：

- 创建资产。
- 提交晋级申请。
- 通过或驳回审批。

事件详情写清操作者、角色、资产、范围变化和结果。事件 UUID 作为本 Demo 的追踪标识。生产版应使用独立不可变审计表，并由服务端补充 IP、会话、租户、策略版本和请求 ID。

### 4.3 一致性边界

审批当前仍是多次 SDK 写入：更新审批、更新资产、写活动事件。任一步失败会显示错误，但不能提供事务原子性。生产版应把它收敛为数据库函数或后端事务；本次不增加自研 API 服务。

## 5. 大屏实现

### 5.1 独立布局

进入 `cockpit` 时不渲染管理侧栏和管理顶栏，由大屏自己的标题栏提供：

- 三视角切换。
- 数据刷新时间。
- 进入/退出全屏。
- 返回管理系统。

### 5.2 视角数据

- 全局：全部资产。
- 团队：团队资产。
- 个人：个人资产。

三种视角复用同一指标函数，避免出现卡片、图表和列表统计口径不一致。

### 5.3 全屏

通过标准 Fullscreen API 对大屏根节点调用 `requestFullscreen()`，通过 `fullscreenchange` 同步按钮状态。浏览器不支持或拒绝时展示明确错误，不伪造成功状态。

## 6. 视觉实现

- 页面背景：浅蓝灰。
- 容器：白色、细蓝灰边框、轻阴影。
- 主色：深企业蓝；健康态使用青绿色；告警使用琥珀；高风险使用红色。
- 字体：系统无衬线字体，数字使用等宽数字特性。
- 大屏使用 CSS Grid，1080P 为基线，2K/4K 增大间距和字号；窄于 1440px 时降低列间距但不隐藏核心信息。
- 不依赖远程字体，避免客户网络环境阻塞首屏。

## 7. 决策与放弃方案

### 7.1 采用集中式前端权限策略

原因：当前任务是客户演示，能够用最小改动让所有页面使用一致规则，并可通过矩阵测试证明。放弃在每个组件里写独立条件，因为容易产生规则漂移。

### 7.2 不新增治理表

原因：原 PRD 的权威对象仍是资产、审批和活动。本次用现有数据证明流程，不为展示卡片增加无法维护的第二套数据。生产级审计表留到有真实身份和服务端执行上下文时建设。

### 7.3 不把 Demo 角色切换包装成真实鉴权

原因：匿名 Key 下的浏览器直连无法形成生产安全边界。页面会明确标注“演示身份”，技术文档明确生产落点为企业身份 + RLS/服务端策略。

## 8. 失败处理

- 权限不允许：前端不执行写入，并给出具体原因。
- InsForge 读取失败：保留页面框架，展示错误，不回退到本地假数据。
- 活动留痕失败：本次操作返回失败，避免页面宣称已完整完成。
- 全屏失败：保持普通大屏模式，提示浏览器拒绝全屏。
- 配置缺失：沿用现有启动错误，明确缺少运行时变量。

## 9. 验证

1. 权限矩阵单元测试覆盖四角色 × 三范围，以及两级审批。
2. 治理指标测试覆盖空数据、风险资产、待审批和三视角。
3. 回归现有指标测试。
4. 运行 `npm test`、`npm run typecheck`、`npm run lint`、`npm run build`。
5. 真实浏览器检查管理页、权限切换、审批、大屏视角和全屏。
6. 发布后核对 Rainbond 构建事件、五组件状态、线上资源与 InsForge 查询。

## 10. 涉及面与环境变量

- 前端：会影响。
- InsForge 写入行为：会影响，新增活动事件写入。
- 数据库 Schema：不影响。
- 上下游协议、消息任务、存储、CI：不影响。
- 文档和测试：会影响。
- **无环境变量变更**，无需修改 `.env.example`、Rainbond Secret、Compose、CI 或运维配置，也无需重启后端或执行数据迁移。
